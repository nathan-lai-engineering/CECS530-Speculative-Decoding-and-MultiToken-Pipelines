\# KV-Cache Strategy



\## KV-Cache Structure

The system maintains two states:

\- committed\_tokens

\- speculative\_tokens



\## Speculative Writes

Tokens generated by the draft model are stored in speculative\_tokens.



\## Verification

Speculative tokens are compared against target model outputs to determine the accepted prefix.



\## Commit

Accepted tokens are moved from speculative\_tokens into committed\_tokens.



\## Rollback

All speculative tokens are cleared after verification.



\## Versioning Scheme

The KV-cache is divided into committed and speculative states.  

Committed tokens represent verified outputs, while speculative tokens represent temporary predictions.  

This acts as a simple versioning mechanism where speculative tokens are discarded on mismatch.



\## Consistency Guarantee

Only tokens that match the target model are committed.  

This ensures that the final output is identical to baseline autoregressive decoding.



\## Memory Overhead

Speculative tokens introduce temporary memory overhead proportional to the speculation depth k.
